# Архитектура

## Общая структура

Приложение делится на следующие части:

- данные словарей;
- настройка тренировки;
- тренировка;
- общие UI-компоненты;
- инфраструктурный код.

## Принципы

- UI не должен содержать в себе доменную логику.
- Алгоритм показа и повторного появления слов должен быть отделён от визуальных компонентов.
- Данные словарей должны быть отделены от логики тренировки.
- Настройки пользователя должны быть отделены от данных словарей и от UI.
- Модули должны быть небольшими и иметь понятный публичный интерфейс через `index.ts`.

## Слои

### Данные

Слой данных отвечает за:

- хранение предустановленных наборов слов;
- фильтрацию слов по выбранным языкам;
- подготовку данных для тренировки.

Этот слой не должен зависеть от UI.

Слова хранятся в массивах объектов, где ключи объектов — коды языков.

Пример:

```js
[
  {
    en: "food",
    ru: "еда",
    es: "comida"
  }
]
```

Надо слов должны запрашиваться асинхронно из внутреннего сервиса. Пока что это
лишь симуляция, но впоследствии должно быть легко перейти на запросы сервера.

### Доменная логика

Доменный слой отвечает за:

- правила тренировки;
- состояние карточек;
- выбор следующего слова;
- регулирование частоты повторного показа слов.

Этот слой не должен зависеть от React-компонентов и визуальных эффектов.

### UI

UI-слой отвечает за:

- экран настройки;
- экран тренировки;
- карточки;
- темы;
- анимации и жесты.

UI использует доменную логику, но не определяет её.

### Инфраструктура

Инфраструктурный слой отвечает за:

- локальное хранение пользовательских настроек, если оно добавлено;
- адаптеры доступа к данным;
- вспомогательные технические утилиты.

## Инварианты

- Источник истины для слов — предустановленные клиентские данные.
- В тренировку попадают только слова, у которых есть значения для обоих выбранных языков.
- Тренировка бесконечна и завершается только явным выходом пользователя.
- Хорошо знакомые слова должны появляться реже.
- Незакреплённые слова должны появляться чаще.
- Одновременно на экране находятся три текущие карточки.

## Правила модулей

- Каждый модуль публикует наружу только свой публичный интерфейс через `index.ts`.
- Чтобы понять, как использовать модуль, должно быть достаточно прочитать его `index.ts`.
- Внутренние файлы модуля не должны импортироваться снаружи напрямую.
- Если модуль разрастается, его нужно дробить, сохраняя понятный публичный интерфейс.

## Направление зависимостей

Допустимое направление зависимостей: данные → доменная логика → UI.
Инфраструктурный код может обслуживать другие слои, но не должен смешивать их ответственности.

## Изменение архитектуры

При добавлении новой функциональности нужно:

- встраивать её в существующие слои;
- не смешивать данные, доменную логику и UI;
- не ломать публичные интерфейсы модулей без необходимости;
- не добавлять новые архитектурные сущности без явной пользы.
