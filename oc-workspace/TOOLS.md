# TOOLS.md

## Рабочие пути

- Репозиторий продукта: `~/.openclaw/workspace/langley`
- Артефакты системы: `~/.openclaw/workspace/artifacts`
- Главная концепция продукта: `~/.openclaw/workspace/langley/README.md`
- Краткосрочная память системы: `~/.openclaw/workspace/artifacts/README.md`

Во всей практической работе я исхожу из этих путей и не ищу проект в других местах,
если только это явно не указано в артефактах или в структуре репозитория.

## Источники истины

Приоритет источников такой:
1. фактическое состояние репозитория;
2. `README.md` в корне репозитория;
3. актуальные артефакты в `workspace/artifacts/actual`;
4. `workspace/artifacts/README.md` как оперативный статус и план.

Если источники противоречат друг другу, я считаю приоритетным более высокий источник
и фиксирую расхождение в артефакте.

## Работа с репозиторием

Вся разработка ведётся внутри `workspace/langley`.

Перед началом технической задачи я обязан:
- перейти в репозиторий;
- прочитать `README.md`;
- проверить структуру проекта;
- определить фактические команды сборки, запуска и тестирования по файлам проекта.

Я не считаю стек, тестовый раннер или команды заранее известными,
пока не проверю репозиторий.

## Git

Контроль версий ведётся через локальный `git` (без GitHub).

Базовые правила:
- не работать с кодом вне git-репозитория;
- перед изменениями смотреть текущее состояние ветки;
- не смешивать несколько независимых задач в один набор изменений;
- не делать изменений в `main` напрямую;
- каждая задача по коду должна жить в отдельной ветке;
- перед завершением фиксировать, какие файлы изменены и зачем.

## Команды проекта

В системе доступны `node` и `npm`.

Но конкретные команды проекта я определяю по самому репозиторию:
- по `package.json`;
- по конфигам тестов, линтеров и сборки;
- по существующим scripts и документации.

Я не устанавливаю новые зависимости без причины.
Сначала использую то, что уже предусмотрено проектом.

## Проверка изменений

Если задача затрагивает код, перед завершением я обязан выполнить
минимально достаточную проверку результата.

Приоритет проверок такой:
1. точечные тесты, относящиеся к изменению;
2. затем lint/typecheck, если они есть;
3. затем более широкий прогон, если это оправдано масштабом изменений.

Я не запускаю тяжёлые команды без необходимости,
но и не завершаю кодовую задачу без проверки.

Если проверка не прошла, я фиксирую:
- какую команду запускал;
- что именно сломалось;
- это блокер или ожидаемое следствие незавершённой работы.

## Работа с артефактами

Артефакты находятся в `workspace/artifacts`:
- `actual` — материалы по текущему фокусу;
- `backlog` — отложенное, но полезное;
- `archive` — утратившее актуальность или завершённое.

Перед вызовом субагента релевантный контекст должен быть в `actual`.
После завершения задачи результат должен быть отражён в артефакте.

Артефакт должен быть прикладным:
- решение;
- спецификация;
- декомпозиция;
- отчёт по реализации;
- отчёт по тестам;
- описание блокера.

Пустые рассуждения без практического вывода не считаются полезным артефактом.

## Практический режим работы

Если задача продуктовая или исследовательская,
мой результат — качественный артефакт для следующего шага.

Если задача инженерная,
мой результат — либо изменения в репозитории с проверкой,
либо точный артефакт с причиной, почему продолжать нельзя.

Каждое действие должно приближать проект к следующему наблюдаемому состоянию:
- новый артефакт;
- обновлённый план;
- готовая задача;
- изменение в коде;
- проверенный результат.
